Introduction

The memorandum "Adoption of the National Information Exchange Model within the Department of Defense" (March 28, 2013) calls for the adoption of a NIEM-based approach, an approach that facilitates the deployment of standards-based XML data exchanges. Therefore, the development of new SOI-provided XML data types shall follow a NIEM-based approach. Major revisions to existing data types should also follow the NIEM-based approach, when applicable and practical.

Therefore, the TSOA approach to developing new SOI-provided data types is largely proscribed by NEIM specifications and guidance. This approach requires the implementation of foundational NIEM data components and encourages the reuse of data components already described by the NIEM community.

The TSOA approach also strives to produce a product that other software developers can deploy, exercise, and reuse. For example, beyond creating static XML Schema documents, this approach implements the following:

  • [ Git ] repository
  • [ Apache Ant ] and Maven test, build, and packaging automation
  • [ W3C XSL 3.0 ] XML and JSON format transformations
  • ISO-19757 Document Schema Definition Languages (DSDL) [ Schematron rule-based validation ]

Furthermore, the standardized TSOA approach encourages software developers to also deploy, exercise, and reuse a common test and build environment which is supported by automated test, build, and package tasks.

  • [ Identify and swap ] data components with NIEM data components, where possible
  • [ Encode and adapt ] XML Schema data component representations in accordance to [ NIEM guidelines ]
  • Map and generate transformations between a non-NIEM-based datatype and its NIEM-based datatype
  • Validate XML Schema representations and NIEM-conformance
  • [ Generate and validate ] NIEM Information Exchange Package (IEP) artifacts, to include exchange samples
  • Package and distribute changes to managed repositories, to include Software Forge.mil at project [ USMC - MAGTF C2 ] SVN repo [ SoiMessaging ].

End-Product

The end-product goal is a NIEM-conformant data type package and common test and build environment that may be deployed, exercised, and re-used by interested software developers. Along with the XML Schema, an end-point may implement certain components directly, such as rule-based validation and data transformation instructions. Other components are readily available for analysis and troubleshooting, and further development and testing.

Maintenance

A data type and its packaged components should be continually updated and improved to maintain their usability and relevancy. Towards this premise, SOI-provided XML data types and their packaged components shall be maintained.
